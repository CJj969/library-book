## D. 建议优化项

### O1：CK_Reservation_Status 检查约束与"已完成"实际存储策略对齐

**位置：** 第 9.1 节，CK_Reservation_Status 约束。

**问题：**  
第 6.2 节明确说明"`已完成` 不做批量 UPDATE，而是每次查询时通过 `ReserveDate < Today AND Status = N'已预约'` 条件在内存中标记"——即 DB 中永远不会出现 `Status = N'已完成'` 的记录。但在第 9.1 节 CHECK 约束中，`Status IN (N'已预约', N'已取消', N'已完成')` 包含了"已完成"。

虽然这不影响功能正确性（EF Core Code-First 默认不生成 CHECK 约束，这个约束即使不实现也不会出问题），但如果真要在迁移脚本中手写 CHECK 约束，它会允许写入"已完成"但实际并没有代码会写它。

**建议：** 有两个选项——

- 选项 A（推荐）：在文档中注明"CK 约束用于文档说明完整性，EF Core Code-First 不会自动生成，后续迁移脚本中可不实现"
- 选项 B：从 CK 约束中移除"已完成"，实际只保留"已预约"和"已取消"

### O2：跨时段过期判定

**位置：** 第 6.2 节，"已完成"判定逻辑。

**问题：**  
当前"已完成"的判定条件是 `ReserveDate < Today`，即只按日期判断。这意味着：如果现在是 2026-07-14 下午 3 点，用户 A 有一个今天"上午"的预约，UI 上仍会显示为"已预约"（带取消按钮），而不是"已完成"。

严格来说它已经过期了（上午时段已过），但当前逻辑不会将其识别为已完成。虽然这不算严重问题（用户点了取消也会失败），但会让"我的预约"页面在下午时段显示上午的预约仍可操作，体验不够准确。

**建议：** 在"已完成"的判定中增加时段维度的比较：

```
已完成 = (ReserveDate < Today) || (ReserveDate == Today AND TimeSlot 已过当前时间)
```

时段顺序：上午(08~12) → 下午(13~17) → 晚上(18~22)。如果当前时间是 14:00，则当日的"上午"预约应标记为已完成，"下午"和"晚上"仍为已预约。

这个改动很小（只在 ViewModel 组装时加一个 `if` 判断），但对用户体验有明显改善。
