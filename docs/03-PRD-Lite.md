# 图书馆座位预约系统 —— PRD Lite

## 1. 项目概述

| 字段 | 内容 |
|------|------|
| 项目名称 | 图书馆座位预约系统 |
| 英文名称 | Library Seat Reservation System (LSRS) |
| 项目代号 | SeatBook |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core + Bootstrap |
| 目标平台 | Web（桌面端优先，兼顾平板） |
| 项目周期 | 4 周 |
| 交付方式 | 项目代码推送至 GitHub / Gitee 仓库，提交仓库链接 |

一句话描述：一个轻量级的图书馆座位预约 Web 应用，学生在线查座→预约→取消，管理员后台管理座位与预约数据。

---

## 2. 项目目标

1. 交付一个可运行的 ASP.NET Core MVC Web 应用
2. 学生可以查看座位、预约座位、取消预约
3. 管理员可以管理座位、查看预约记录、查看统计数据
4. 项目代码托管到 GitHub / Gitee，提交可用的仓库链接

---

## 3. 目标用户

| 用户类型 | 身份 | 使用频率 | 核心诉求 |
|----------|------|----------|----------|
| 普通学生 | 在校本科生/研究生 | 每日/隔日 | 快速找到空座并预约 |
| 管理员 | 图书馆工作人员 | 每日 | 管理座位和预约记录 |

普通用户采用体验账号切换（导航栏下拉框选择预设账号），不开发真实注册。  
管理员使用轻量 Session/Cookie 登录，Seed 数据预置一个默认管理员账号。

---

## 4. 使用场景

**场景一：学生预约座位**  
小王晚上想约明天图书馆的座位，打开系统看到三楼有空位，选了一个靠窗座位，预约明天 9:00-12:00，系统提示预约成功。

**场景二：学生取消预约**  
小王第二天起晚了去不了，打开"我的预约"取消今天的预约，座位被释放。

**场景三：管理员管理座位**  
管理员发现 3 楼 12 号座位灯管坏了，在后台标记为"维修中"，学生端不再显示可预约。

**场景四：管理员查看统计**  
期末临近，管理员登录后台查看统计页，看到本周座位平均利用率 85%。

## 5. 用户端核心功能

### F01 — 体验账号切换
- 导航栏右上角下拉框，列出 3~4 个预设体验账号
- 选中即切换当前会话身份，不刷新页面跳转
- 当前登录身份在导航栏可见

**验收标准：**
- [ ] 下拉框列出预设账号名称
- [ ] 选中后页面刷新/局部刷新，身份切换生效
- [ ] 不同身份的"我的预约"数据隔离

### F02 — 首页
- 展示欢迎语、系统公告（静态文字）
- 座位概况卡片（空闲/已约/维修中数量）
- 快速导航入口（进入座位列表、查看我的预约）

**验收标准：**
- [ ] 页面正常渲染，无报错
- [ ] 座位概况数字与数据库一致
- [ ] 导航入口可点击跳转

### F03 — 座位列表
- 按区域/楼层展示座位网格
- 每个座位卡片显示座位号、区域、状态（空闲/已预约/维修中）
- 不同状态使用不同颜色区分
- 顶部提供区域 Tab 切换

**验收标准：**
- [ ] 展示至少 20 个座位，三种状态均有覆盖
- [ ] 状态颜色区分明显（如绿/红/灰）
- [ ] 点击空闲座位可进入详情页

### F04 — 座位详情（含预约表单）
- 显示座位编号、所在区域、座位描述
- 显示该座位的时段占用时间轴（上午/下午/晚上）
- 内嵌预约表单：选择日期 + 选择时段 + 提交按钮
- 提交后校验冲突，冲突时提示不可约

**验收标准：**
- [ ] 空闲座位显示可选的时段
- [ ] 已约时段置灰不可选
- [ ] 提交后若冲突则提示"该时段已被预约"
- [ ] 提交成功则 TempData 提示并跳转至"我的预约"

### F05 — 我的预约
- 展示当前用户的所有预约记录（进行中 + 已过期）
- 每条记录显示：座位号、区域、日期、时段、状态
- 可取消的预约显示"取消"按钮
- 已过期/已取消的记录置灰

**验收标准：**
- [ ] 预约成功后的记录立即出现
- [ ] 取消后状态变为"已取消"
- [ ] 已过期预约不显示取消按钮

### F06 — 取消预约
- 在"我的预约"页点击取消按钮
- 弹出确认对话框
- 确认后释放座位，状态变为"已取消"

**验收标准：**
- [ ] 取消后座位在列表中恢复"空闲"状态
- [ ] 取消后预约状态变为"已取消"
- [ ] 已过期预约不可取消

## 6. 管理端核心功能

### F07 — 管理员登录
- 登录页：账号 + 密码表单
- 使用 Session/Cookie 保存登录状态
- 登录后跳转管理后台首页（预约管理页）
- 未登录访问管理端页面自动跳回登录页
- Seed 数据预置默认管理员（admin / 123456）

**验收标准：**
- [ ] 输入正确账号密码可登录
- [ ] 输入错误密码提示"账号或密码错误"
- [ ] 未登录时访问管理端 URL 被拦截

### F08 — 预约管理
- 表格展示所有用户的预约记录
- 筛选条件：按日期、按区域、按状态
- 每行操作：取消预约（强制取消）、标记完成
- 支持分页

**验收标准：**
- [ ] 展示全部预约，数据与用户端一致
- [ ] 筛选功能可用
- [ ] 管理员取消后，用户端对应预约状态更新

### F09 — 座位管理
- 表格/卡片展示所有座位
- 操作：新增座位（编号 + 区域 + 描述）、编辑、删除、启用/禁用
- 禁用后用户端该座位显示"维修中"

**验收标准：**
- [ ] 新增的座位在用户端立即可见
- [ ] 禁用的座位用户端不可预约
- [ ] 删除座位确认后生效

### F10 — 统计页
- 展示预约总数（今日/本周/本月）
- 展示座位利用率（预约时段/总可用时段）
- 至少两个数字指标，用卡片或简单表格展示

**验收标准：**
- [ ] 数字与数据库数据一致
- [ ] 页面无报错

### 共享功能

### F11 — 404 错误页
- 访问不存在 URL 时显示友好提示页面
- 包含返回首页的链接

**验收标准：**
- [ ] 访问 `/nonexistent` 返回 404 页，非白屏

### F12 — Seed 数据初始化
- 项目首次启动/每次启动时自动生成演示数据
- 至少：3 个区域、20 个座位、4 个体验账号（学生 A/B/C + 管理员）

**验收标准：**
- [ ] 首次运行后数据库有预置数据，无需手动插入

## 8. 核心业务流程

### 预约流程
```
用户选择体验账号 → 进入座位列表 → 点击空闲座位
→ 查看座位详情 → 选择日期+时段 → 提交预约
→ 系统校验冲突 → 成功 → TempData 提示 → 跳转"我的预约"
```

### 取消预约流程
```
用户进入"我的预约" → 查看可取消的预约 → 点击取消
→ 弹窗确认 → 确认 → 释放座位 → 状态变"已取消"
```

### 管理员完整操作流程
```
访问管理登录页 → 输入 admin/123456 → 登录成功
→ 进入预约管理（默认页） → 查看/筛选/取消预约
→ 切换到座位管理 → 新增/编辑/禁用座位
→ 切换到统计页 → 查看数据
→ 退出登录
```

## 11. 核心数据对象

### Seat（座位）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int | 主键，自增 |
| SeatNumber | string | 座位编号（如"A-01"） |
| Area | string | 所在区域（如"三楼大厅"） |
| Description | string | 座位描述（如"靠窗、有插座"） |
| IsActive | bool | 是否启用（false = 维修中） |
| CreatedAt | DateTime | 创建时间 |

### Reservation（预约）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int | 主键，自增 |
| SeatId | int | 外键 → Seat |
| UserId | int | 外键 → User（体验账号） |
| ReserveDate | DateTime | 预约日期（仅日期部分） |
| TimeSlot | string | 时段："上午"/"下午"/"晚上" |
| Status | string | 状态："已预约"/"已取消"/"已完成" |
| CreatedAt | DateTime | 预约创建时间 |

### User（体验账号/管理员）

| 字段 | 类型 | 说明 |
|------|------|------|
| Id | int | 主键，自增 |
| Name | string | 显示名称（如"学生 A"） |
| Role | string | 角色："Student"/"Admin" |
| Password | string | 密码（仅管理员有值，学生为空） |

## 12. 状态流转说明

### 座位状态

```
         ┌────────────┐
         │  空闲      │
         └─────┬──────┘
               │ 被预约
               ▼
         ┌────────────┐
         │  已预约    │
         └─────┬──────┘
               │ 取消预约 / 过期
               ▼
         ┌────────────┐
         │  空闲      │
         └────────────┘

管理员手动操作：
         ┌────────────┐
         │  启用      │ ←→ 禁用（维修中）
         └────────────┘
```

### 预约状态

```
 提交预约 → [已预约]
              │
        ┌─────┴─────┐
        ▼           ▼
   [已取消]    [已完成]（管理员标记/自动过期）
```

取消规则：
- 预约时间到达前可取消
- 预约时间到达后不可取消（只能管理员操作或标记过期）
- 取消后座位立即释放

## 15. 风险与假设

| 风险 | 可能性 | 应对措施 |
|------|--------|----------|
| 时间不够，用户端和管理端做不完 | 高 | 先保用户端核心闭环（F01→F10）完整跑通，管理端做最简 CRUD |
| LocalDB 环境配不出来 | 中 | 提前在组内统一 .NET 8 SDK 版本，准备 Docker/SQLite 备选 |
| EF Core 迁移冲突搞乱数据 | 中 | 每次改模型立刻 `add-migration`，不攒一堆；必要时删库重建 |
| AI 生成代码的预约冲突检测有 Bug | 高 | 重点测试：同一座位+同一时段+不同用户能否重复约（应拒绝） |
| 页面超出 9 页控制不住 | 中 | 每次新增页面必须确认是否在立项范围内，超 12 页砍掉尾部功能 |
| 没做 Git 管理，代码丢了 | 中 | 从第一天初始化 Git，每完成一个页面就 commit |
