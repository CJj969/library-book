## A. 总体判断

**质量很高，1 个必改项 + 3 个建议项。修完必改项即可进入下一步。**

这是目前所有交付件中完成度最高的文档。9 张页面卡完整覆盖所有页面，每页的成功/空/异常状态都写清楚了，UI 规范从色彩到字号到组件尺寸颗粒度很细，原型三层次和评审清单也很实用。最大问题是管理端导航配色方案写的是"待定"——这对原型生成是硬阻塞。

## C. 必改项

### P1：管理端导航配色方案"待定"，无法指导原型生成

**位置**：第 5.8 节（管理端导航），第 366~367 行。

**原文：**
> 方案 | 顶部水平导航栏（与用户端一致但链接不同）
> 背景 | `#1E293B`（深色）或 `#FFFFFF`（浅色方案）
> 待定 | 原型时确定深色或浅色方案，推荐浅色保持统一

**问题：**  
"待定"意味着原型师不知道管理端导航栏用深色还是浅色背景，底部管理页面的卡片/表格配色也会跟着动摇（深色导航下表格背景是否需要调整？）。这属于设计决策未关闭。

**修正方案：**

直接锁定推荐方案，移除"待定"：

> 管理端采用**浅色方案**，导航栏样式与用户端保持一致（白色背景 `#FFFFFF`，底部 `1px solid #E2E8F0` 边框，高度 56px，水平内边距 16px），仅导航链接不同（预约管理 / 座位管理 / 统计页 / 退出登录）。
>
> 当前体验账号名称显示在导航栏右侧，"退出登录"按钮紧邻其后。

理由：浅色方案与用户端共享同一套组件规范，减少 CSS 代码量；管理端只需额外定义一个"退出登录"按钮样式即可，不需要独立深色主题。

## D. 建议优化项

### O1：统一管理员默认密码

PRD Lite（第 6 节 F07 验收标准）写的是 `admin / 123456`，本文 PC-05 写的是 `admin / admin123`。两处数据不一致，最终实现时开发者不知道该信哪一个。建议统一为 `admin / 123456`（更短，更符合"临时演示密码"的直觉）。成本：改 PC-05 空状态一行字。

### O2：删除"连续失败可重置"的暗示

PC-05 异常提示写"连续失败 → 提示可重置（无验证码）"。本期范围没有密码重置功能，Seed 只有一条管理员记录，也不存在"重置"的逻辑。这句话会误导开发者去做一个不存在于任何需求文档里的功能。建议改为：

> 连续失败 → 红色提示"账号或密码错误"，无其他操作

### O3：补充全局页面骨架描述

当前页面卡和组件规范分散在各处，但没有一张全局的"页面骨架"图来回答以下问题：

- 用户端所有页面的共同结构是什么？（导航栏 + 内容区 + ？）
- 管理端所有页面的共同结构是什么？（导航栏 + 内容区 + ？）
- 内容区是否有左右边距约束？

建议在 UI 规范中增加一小节（放在组件规范之前），用一句话 + 一个简图描述：

```
用户端页面骨架：
┌──────────────────────────────┐
│  导航栏 (56px)    [体验账号▼] │
├──────────────────────────────┤
│  内容区（最大宽度 480px，居中）│
│                              │
└──────────────────────────────┘

管理端页面骨架：
┌──────────────────────────────┐
│  导航栏 (56px)    [admin▼ 退出] │
├──────────────────────────────┤
│  内容区（全宽，最小 1024px）  │
│                              │
└──────────────────────────────┘
```

当前 9 张页面卡中的"布局要求"字段已足够详细，但这个全局骨架能帮助原型师在动手前快速理解两端的布局差异。

## E. 是否可以进入下一步

**修完 P1 后可以进入"静态原型与原型评审"。**

P1 是唯一的阻塞项——管理端导航配色不定，原型师无法决定导航栏颜色、链接样式、激活态底色，会导致静态原型做到一半停下来等人拍板。锁定浅色方案后，9 张页面卡 + 全套 UI 规范 + 响应式断点已经足够支撑原型生成。

O1~O3 可以在原型制作过程中同步修改，不需要卡住当前进度。
