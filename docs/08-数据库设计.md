# 图书馆座位预约系统 —— 数据库设计

## 1. 数据设计目标

| 目标 | 说明 |
|------|------|
| 够用 | 只用 3 张表支撑 9 页的全部数据和查询需求 |
| 可查 | 关键查询（冲突检测、我的预约、管理端筛选、统计）走索引 |
| 一致 | 外键约束 + 唯一索引保障数据不会出现"一个座位同时被两个人约到" |

**设计原则：** 不因为"以后可能用到"而多加表和字段。3 张表能解决的问题，不拆成 5 张表。

## 2. 核心实体清单

从前序文档（PRD Lite 第 11 节）已确认的实体共 **3 个**，不增不减：

| 实体 | 说明 | 来源 |
|------|------|------|
| **User** | 体验账号 + 管理员，二合一 | PRD Lite §11.3 |
| **Seat** | 座位基本信息和启用状态 | PRD Lite §11.1 |
| **Reservation** | 预约记录，含状态和时段 | PRD Lite §11.2 |

**不需要的实体（未在前序文档中出现）：**
- ❌ `SeatArea` 独立表 — Area 作为 Seat 的字符串字段足够
- ❌ `TimeSlot` 独立表 — 时段固定为 3 个值，硬编码校验即可
- ❌ `ReservationLog` — 本期不做操作审计日志

## 3. 关系说明

```
User (1) ──────< (N) Reservation (N) >────── (1) Seat
  │                                            │
  ├─ 一个用户可以有多条预约                   ├─ 一个座位可以被多条预约引用
  └─ 一条预约属于一个用户                     └─ 一条预约引用一个座位
```

**关系约束：**
- Reservation.UserId → User.Id（外键，CASCADE 删除不启用 — 体验账号不删除）
- Reservation.SeatId → Seat.Id（外键，CASCADE 删除不启用 — 座位有预约时不允许删除）
- 无多对多关系，无自引用关系

## 4. 表结构设计

### 4.1 User（用户/体验账号）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| Id | `int` | PK, 自增 | 主键 |
| Name | `nvarchar(50)` | NOT NULL | 显示名称，如"学生 A"、"管理员" |
| Role | `nvarchar(20)` | NOT NULL | 角色：`Student` 或 `Admin` |
| Password | `nvarchar(100)` | NULL | 仅管理员有值；学生账号为 NULL |

**Seed 数据（6 条）：**

| Name | Role | Password |
|------|------|----------|
| 学生 A | Student | NULL |
| 学生 B | Student | NULL |
| 学生 C | Student | NULL |
| 学生 D | Student | NULL |
| 管理员 | Admin | 123456 |

**索引：**
- PK: `IX_User_Id`（默认聚集索引）

### 4.2 Seat（座位）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| Id | `int` | PK, 自增 | 主键 |
| SeatNumber | `nvarchar(20)` | NOT NULL, UNIQUE | 座位编号，如 `A-01`、`B-12` |
| Area | `nvarchar(50)` | NOT NULL | 所在区域，如 `2楼A区`、`3楼大厅` |
| Description | `nvarchar(200)` | NULL | 描述，如 `靠窗、有插座` |
| IsActive | `bit` | NOT NULL, DEFAULT 1 | 是否启用：`1`=启用, `0`=维修中 |
| CreatedAt | `datetime2` | NOT NULL, DEFAULT GETDATE() | 创建时间 |

**Seed 数据（约 20 条）：**
建议分布在 3 个区域：2楼A区（6个）、2楼B区（6个）、3楼A区（8个）。编号示例：`A-01`~`A-06`、`B-01`~`B-06`、`C-01`~`C-08`。其中 2~3 个座位 `IsActive = 0`（维修中）。

**索引：**
- PK: `IX_Seat_Id`
- UK: `IX_Seat_SeatNumber`（唯一索引，SeatNumber）
- IX: `IX_Seat_Area`（非唯一，区域查询）

### 4.3 Reservation（预约记录）

| 字段名 | 类型 | 约束 | 说明 |
|--------|------|------|------|
| Id | `int` | PK, 自增 | 主键 |
| SeatId | `int` | FK → Seat.Id, NOT NULL | 预约的座位 |
| UserId | `int` | FK → User.Id, NOT NULL | 发起预约的用户 |
| ReserveDate | `date` | NOT NULL | 预约日期（仅日期，无时间） |
| TimeSlot | `nvarchar(10)` | NOT NULL | 时段：`上午` / `下午` / `晚上` |
| Status | `nvarchar(10)` | NOT NULL, DEFAULT `已预约` | 状态：`已预约` / `已取消` / `已完成` |
| CreatedAt | `datetime2` | NOT NULL, DEFAULT GETDATE() | 预约创建时间 |

**字段说明：**
- `ReserveDate` 只存日期部分（如 `2026-07-15`），不存时间。`date` 类型比 `datetime` 更精确表达"哪天"。
- `TimeSlot` 固定为 3 个值，不做分钟级自定义时段。三者的映射关系：

| TimeSlot 值 | 对应时间段 |
|-------------|-----------|
| 上午 | 08:00 ~ 12:00 |
| 下午 | 13:00 ~ 17:00 |
| 晚上 | 18:00 ~ 22:00 |

- `Status` 使用中文字符串值，与 PRD Lite §11.2 一致。三个值及含义：

| Status 值 | 含义 | 触发条件 |
|-----------|------|----------|
| 已预约 | 预约有效，座位被占用 | 提交预约成功后 |
| 已取消 | 预约无效，座位已释放 | 用户手动取消 / 管理员强制取消 |
| 已完成 | 预约过期，座位已释放 | 系统按日期自动判定（查询时逻辑判断） |

**索引：**

| 索引名 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| PK_Reservation_Id | Id | 聚集 | 主键 |
| IX_Reservation_UserId | UserId | 非唯一 | 加速"我的预约"查询 |
| IX_Reservation_SeatDateSlot | SeatId, ReserveDate, TimeSlot | 非唯一 | 加速冲突检测 |
| IX_Reservation_Status | Status | 非唯一 | 加速管理端按状态筛选 |
| IX_Reservation_ReserveDate | ReserveDate | 非唯一 | 加速按日期统计和筛选 |
| IX_Reservation_UserId_Status | UserId, Status | 非唯一 | 加速"我的预约"按状态过滤 |

**唯一约束（关键防冲突）：**

```sql
-- 保证同一座位同一天同一时段最多只有一条"已预约"记录
CREATE UNIQUE NONCLUSTERED INDEX UQ_Reservation_Active
ON Reservation (SeatId, ReserveDate, TimeSlot)
WHERE Status = N'已预约';
```

这个筛选唯一索引（Filtered Unique Index）是防止重复预约的核心保障。它只对 `Status = '已预约'` 的记录生效，`已取消` 和 `已完成` 的记录不受此约束。

## 5. 关键字段说明

| 字段 | 为什么这样设计 | 替代方案及排除理由 |
|------|---------------|-------------------|
| `Seat.IsActive`（bit） | 座位只有"启用/维修中"两个状态。"是否被占"是动态数据，不存 DB | ❌ 存 `Status` 枚举（`Available/Reserved/Maintenance`）：会导致每次预约/取消时需要同时更新 Seat 表，增加一致性问题 |
| `Reservation.ReserveDate`（date） | 只关心"哪天"，不关心"几点几分预约的" | ❌ 用 `datetime` 存日期+时间：查询时需要 `CAST` 去除时间部分，索引效率下降 |
| `Reservation.TimeSlot`（string） | 固定 3 个时段，字符串精确匹配即可查冲突 | ❌ 用 `StartTime/EndTime` TimeSpan：冲突检测需范围查询（`start1 < end2 AND start2 < end1`），SQL 更复杂、索引更难。对 MVP 来说固定时段足够 |
| `Reservation.Status`（string） | 中文字段值与 PRD Lite 一致，开发者读 DB 时一目了然 | ❌ 用 `int` 枚举：需要 Join 字典表才能在 SSMS 中读懂 |
| `User.Password`（nullable） | 学生账号无密码，管理员有密码 | ❌ 拆成两张表（Student / Admin）：增加 Join 成本，本期学生端只有 4 个固定账号 |

## 6. 状态字段设计

### 6.1 座位状态（Seat.IsActive）

```
IsActive = 1 (启用)
    ↑ 管理员操作 ↓
IsActive = 0 (维修中)
```

> 座位"是否被预约"不是状态字段，而是通过查询 Reservation 表实时计算。计算逻辑：
> `SELECT 1 FROM Reservation WHERE SeatId = @id AND ReserveDate = @today AND TimeSlot = @slot AND Status = N'已预约'`

### 6.2 预约状态（Reservation.Status）

```
         ┌──────┐
         │ 已预约 │  ← 提交预约成功后
         └───┬──┘
             │
    ┌────────┼────────┐
    │        │        │
 用户主动  管理员    日期已过（系统查询时
 取消      强制取消  自动判定为"已完成"）
    │        │        │
 ┌──▼──┐  ┌──▼──┐  ┌──▼──┐
 │已取消│  │已取消│  │已完成│
 └─────┘  └─────┘  └─────┘
```

**状态变更入口：**

| 变更 | 触发动作 | 代码位置 |
|------|----------|----------|
| → 已预约 | `ReservationService.CreateReservation()` | Service 层 |
| 已预约 → 已取消 | `ReservationService.CancelReservation()` | Service 层 |
| 已预约 → 已完成 | 查询时逻辑判断（不 UPDATE DB） | 查询时 WHERE 条件或 ViewModel 组装中判断 |

> `已完成` 不做批量 UPDATE，而是每次查询时通过条件在内存中标记。判定逻辑：
> ```
> 已完成 = (ReserveDate < Today) || (ReserveDate == Today && TimeSlot 已过当前时间)
> ```
> 时段顺序：上午(08~12) → 下午(13~17) → 晚上(18~22)。例如当前时间 14:00，则当日的"上午"预约标记为已完成，"下午""晚上"仍为已预约。
> 
> 理由是：不需要每天跑定时任务，减少复杂度。

### 6.3 页面表现映射

| DB 状态值 | 页面显示标签 | 标签色值 | 操作按钮 |
|-----------|-------------|----------|----------|
| 已预约 | "已预约" | `#2563EB` 蓝底 | 显示"取消预约"按钮 |
| 已取消 | "已取消" | `#DC2626` 红底 | 无操作按钮 |
| 已完成 | "已完成" | `#059669` 绿底 | 无操作按钮 |

## 7. 查询与筛选字段设计

### 7.1 高频查询一览

| 查询场景 | 涉及表 | WHERE 条件 | 使用的索引 |
|----------|--------|-----------|-----------|
| 冲突检测 | Reservation | `SeatId=@s AND ReserveDate=@d AND TimeSlot=@t AND Status=N'已预约'` | IX_Reservation_SeatDateSlot + UQ_Reservation_Active |
| 我的预约 | Reservation + Seat | `UserId=@uid` | IX_Reservation_UserId |
| 管理端预约列表 | Reservation + Seat + User | `Status=@s`（可选） | IX_Reservation_Status |
| 按日期筛选预约 | Reservation | `ReserveDate BETWEEN @d1 AND @d2` | IX_Reservation_ReserveDate |
| 按区域查座位 | Seat | `Area=@area` | IX_Seat_Area |
| 座位列表（含状态） | Seat + Reservation | LEFT JOIN 实时计算 | IX_Reservation_SeatDateSlot |

### 7.2 EF Core 查询示例

**冲突检测：**

```csharp
var conflict = await _context.Reservations.AnyAsync(r =>
    r.SeatId == seatId &&
    r.ReserveDate == reserveDate &&
    r.TimeSlot == timeSlot &&
    r.Status == "已预约");
```

**我的预约（关联座位信息）：**

```csharp
var list = await _context.Reservations
    .Include(r => r.Seat)
    .Where(r => r.UserId == userId)
    .OrderByDescending(r => r.CreatedAt)
    .ToListAsync();
```

**座位列表（附带当日各时段占用状态）：**

```csharp
var seats = await _context.Seats.ToListAsync();

// 这里不直接在 SQL 中算占用状态，而是加载后组装
var todayReservations = await _context.Reservations

// 在内存中组装 SeatListViewModel
```

## 8. 统计字段与统计口径

### 8.1 统计指标及 SQL 口径

| 指标 | 计算方式 | 对应 SQL / LINQ |
|------|----------|-----------------|
| 总座位数 | `COUNT(Seat)` | `_context.Seats.Count()` |
| 可用座位数 | `COUNT(Seat) WHERE IsActive=1` | `_context.Seats.Count(s => s.IsActive)` |
| 总预约数 | `COUNT(Reservation)` | `_context.Reservations.Count()` |
| 今日预约数 | `COUNT(Reservation) WHERE ReserveDate=Today` | `_context.Reservations.Count(r => r.ReserveDate == today)` |
| 今日已预约座位数 | `COUNT(DISTINCT SeatId) WHERE ReserveDate=Today AND Status=N'已预约'` | `_context.Reservations.Where(r => r.ReserveDate == today && r.Status == "已预约").Select(r => r.SeatId).Distinct().Count()` |
| 座位利用率 | 今日已预约座位数 ÷ 总可用座位数 × 100% | 以上两个值相除 |

### 8.2 统计实现位置

所有统计在 `StatisticsService` 中实现，通过调用 `_context` 进行聚合查询。原则：

- 不建统计表（不冗余存储统计数据）
- 不建物化视图（LocalDB 不支持）
- 每次页面加载实时计算（数据量小，3 张表，查询毫秒级完成）

### 8.3 EF Core 统计代码模式

```csharp
public class StatisticsService : IStatisticsService
{
    private readonly AppDbContext _context;

    public HomePageStats GetHomePageStats()
    {
        var today = DateTime.Today;

        return new HomePageStats
        {
            TotalSeats = _context.Seats.Count(),
            AvailableSeats = _context.Seats.Count(s => s.IsActive),
            ReservedToday = _context.Reservations
                .Count(r => r.ReserveDate == today && r.Status == "已预约")
        };
    }

    public AdminStats GetAdminStats()
    {
        var today = DateTime.Today;
        var totalActiveSeats = _context.Seats.Count(s => s.IsActive);
        var reservedTodaySeats = _context.Reservations
            .Where(r => r.ReserveDate == today && r.Status == "已预约")
            .Select(r => r.SeatId)
            .Distinct()
            .Count();

        return new AdminStats
        {
            TotalSeats = _context.Seats.Count(),
            TotalReservations = _context.Reservations.Count(),
            TodayReservations = _context.Reservations
                .Count(r => r.ReserveDate == today),
            UtilizationRate = totalActiveSeats > 0
                ? Math.Round((double)reservedTodaySeats / totalActiveSeats * 100, 1)
                : 0
        };
    }
}
```

## 9. 数据一致性与约束说明

### 9.1 数据库层级约束

| 约束类型 | 约束内容 | 作用 |
|----------|----------|------|
| FK_Reservation_SeatId | Reservation.SeatId → Seat.Id | 防止预约不存在的座位 |
| FK_Reservation_UserId | Reservation.UserId → User.Id | 防止预约不存在的用户 |
| UQ_Reservation_Active | (SeatId, ReserveDate, TimeSlot) WHERE Status=N'已预约' | 防止重复预约（核心防冲突） |
| UK_Seat_SeatNumber | Seat.SeatNumber 唯一 | 防止座位编号重复 |
| CK_Reservation_TimeSlot | TimeSlot IN (N'上午', N'下午', N'晚上') | 防止写入无效时段 |
| CK_Reservation_Status | Status IN (N'已预约', N'已取消', N'已完成') | 防止写入无效状态（"已完成"仅用于文档完整性说明，实际代码中不写入 DB，通过查询时逻辑判定） |
| CK_User_Role | Role IN (N'Student', N'Admin') | 防止写入无效角色 |
| DF_Seat_IsActive | IsActive DEFAULT 1 | 新增座位默认启用 |
| DF_Reservation_Status | Status DEFAULT N'已预约' | 新增预约默认已预约 |
| DF_Reservation_CreatedAt | CreatedAt DEFAULT GETDATE() | 自动记录创建时间 |

### 9.2 应用层约束（Service 层补充）

数据库约束不能覆盖的校验，在 Service 层实现：

| 校验 | 说明 | 位置 |
|------|------|------|
| 座位是否存在 | EF Core Find 或 FirstOrDefault | 每个操作前 |
| 预约是否属于当前用户 | 取消时校验 UserId 匹配 | `CancelReservation()` |
| 预约日期是否过去 | `ReserveDate < DateTime.Today` 时拒绝提交 | `CreateReservation()` |
| 预约是否已过期不可取消 | 今日之前的预约不可取消；今日的预约若时段已过也不可取消 | `CancelReservation()` |
| 删除座位时检查关联预约 | 有预约记录（不含已取消）的座位不可删除 | `DeleteSeat()` |

### 9.3 软删除策略

- Reservation 不做物理删除，通过 `Status` 字段标记取消
- Seat 不做物理删除，通过 `IsActive = 0` 标记维修中；仅当该座位没有任何预约记录时才可真正 DELETE

## 10. 当前阶段实现边界

### 在数据库范围内

| 事项 | 做法 |
|------|------|
| 表数量 | 3 张（User / Seat / Reservation） |
| 迁移方式 | EF Core Code-First Migration（`Add-Migration` + `Update-Database`） |
| Seed 数据 | `AppDbContext.OnModelCreating` 中通过 `HasData()` 配置，或 `SeedData.cs` 中 `EnsureCreated` 后插入 |
| 数据类型 | `nvarchar` / `int` / `bit` / `date` / `datetime2`，不使用 SQL Server 特有类型（如 hierarchyid） |

### 不在数据库范围内

| 事项 | 说明 |
|------|------|
| 存储过程 | 本期不写 SP，全部通过 LINQ 查询 |
| 视图 | 本期不建 View，ViewModel 在 C# 层组装 |
| 触发器 | 本期不使用 Trigger |
| 索引调优 | 当前索引设计覆盖全部高频查询，不提前做性能优化 |
| 数据库迁移脚本 | 使用 EF Core 自动迁移，不手动写 SQL |

## 11. 默认补足项 / 当前假设

前序文档（PRD Lite §11、系统设计说明 §3）已确认实体为 3 个，未使用默认起点中的 6 实体方案，因此无默认补足项。

**仅记录以下设计决策：**

| 决策点 | 前序文档依据 | 实际选择 |
|--------|-------------|----------|
| 时段表示 | PRD Lite §11.2: `TimeSlot` (string) | ✅ 采用 `TimeSlot`，固定 3 值 |
| 座位状态 | PRD Lite §11.1: `IsActive` (bool) | ✅ 采用 `IsActive`，不单独建状态枚举表 |
| 预约完成判断 | 系统设计 §6: 查询时逻辑判断 | ✅ 不做定时任务 UPDATE 状态 |
| 唯一索引防冲突 | PRD Lite §8: 冲突检测要求 | ✅ `UQ_Reservation_Active` 筛选唯一索引 |
| 时段对应时间映射 | 前序文档未指定具体时间 | 补充：上午 08~12 / 下午 13~17 / 晚上 18~22 |
