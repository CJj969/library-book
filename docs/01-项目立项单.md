# 图书馆座位预约系统 —— 项目立项单

## 1. 项目基本信息

| 字段 | 内容 |
|------|------|
| 项目名称 | 图书馆座位预约系统 |
| 英文名称 | Library Seat Reservation System (LSRS) |
| 项目代号 | SeatBook |
| 技术栈 | ASP.NET Core MVC (.NET 8) + Razor + SQL Server LocalDB + EF Core + Bootstrap |
| 目标平台 | 用户端手机优先，管理端桌面优先 |
| 项目周期 | 4 周 |
| 课程阶段 | 第三阶段 —— 独立项目实战 |
| 交付方式 | 项目代码推送至 GitHub / Gitee 仓库，提交仓库链接 |

## 2. 项目背景

学校图书馆高峰期"一座难求"是长期痛点。现有的占座方式效率低、不公平，经常出现"人不在，书在座"的情况。之前已经有不少学长尝试做过类似系统，但大多半途而废。这次我们的定位非常务实：**不做大而全，只做最小可用**，先把"查座 — 选座 — 预约 — 取消"这个闭环跑通。

这也是你入职后第一个独立负责的小项目，做完之后要能真正跑起来给人看，而不是一堆代码躺在硬盘里。

## 3. 目标用户

- **普通学生**：需要预约图书馆座位、查看自己预约、取消预约
- **管理员（图书馆工作人员）**：管理座位、管理预约记录、查看统计数据

本期不做真实注册登录，普通用户采用体验账号切换或轻量模拟登录，管理员用简单 Session/Cookie 登录。

体验账号切换方式：在全局导航栏右上角设置下拉框，内置 5 个预设体验账号（学生 A / 学生 B / 学生 C / 学生 D / 管理员），选中即切换当前会话身份，无需密码。

## 4. 项目目标

1. 交付一个可运行的 ASP.NET Core MVC Web 应用
2. 学生可以在线预约和取消座位
3. 管理员可以管理座位和预约记录
4. 项目代码托管到 GitHub / Gitee，提交可用的仓库链接

## 5. 本期必须完成内容

以下功能全部实现才算完工：

### 通用要求
- 体验账号切换：导航栏右上角下拉框，内置 5 个预设账号（学生 A/B/C/D + 管理员），选中即切换身份

### 用户端
- 首页：展示系统公告、座位概况、快速导航
- 座位列表：按楼层/区域查看座位，展示占用状态
- 座位详情：查看单个座位的详细信息、时段占用情况，内嵌预约表单（选择时段 → 提交预约）
- 我的预约：查看自己名下的所有预约记录，支持取消预约

### 管理端
- 管理员登录（轻量 Session/Cookie）
- 预约管理：查看所有预约、筛选、取消违规预约
- 座位管理：添加、编辑、删除座位，开关座位状态
- 统计页：展示预约量、座位利用率等基础统计数据

## 6. 本期暂不开发内容

以下功能明确不做，不要在上面花时间：

- ❌ 支付与押金
- ❌ 扫码签到/签退
- ❌ 消息推送（短信/邮件/App 推送）
- ❌ 真实注册/账号体系（用体验账号代替）
- ❌ 多馆区联动
- ❌ 实时座位传感器/硬件对接
- ❌ 移动端 App（响应式 Web 即可）

## 7. 页面规模与页面分布

总页面数约 **9 页**，分布如下：

| 模块 | 页面 | 数量 |
|------|------|------|
| 用户端 | 首页、座位列表、座位详情（含预约表单）、我的预约 | 4 |
| 管理端 | 管理员登录、预约管理、座位管理、统计页 | 4 |
| 共享 | 404 / 错误页 | 1 |

## 8. 关键业务主流程

### 预约流程
```
学生访问首页 → 进入座位列表 → 选择空闲座位 → 查看座位详情
→ 选择预约时段 → 提交预约 → 系统校验 → 预约成功 → 跳转"我的预约"
```

### 取消预约流程
```
学生进入"我的预约" → 查看可取消的预约 → 点击取消 → 确认取消
→ 系统释放座位 → 取消成功
```

### 管理员管理流程
```
管理员登录 → 进入管理后台 → 选择管理模块 → 执行操作 → 退出登录
```

## 9. 成功标准

以下标准需要在最终演示时逐条验证：

1. 项目启动时通过 Seed 自动生成至少 3 个区域、20 个座位和 1 个管理员账号，无需手动插数据
2. 在导航栏下拉框切换体验账号，切换后身份立即生效
3. 启动项目后能在浏览器看到首页
4. 从首页能点进座位列表，看到座位占用状态
5. 点击空闲座位进入详情页，填写时段后提交预约，提示成功
6. 预约后在"我的预约"中能看到该记录
7. 能成功取消预约，取消后座位恢复空闲
8. 管理员能登录后台
9. 管理员能看到所有预约记录，并能手动取消
10. 管理员能新增/编辑/删除座位
11. 统计页能显示基础的预约数量和座位利用率数字

## 10. 风险与假设

| 风险 | 应对措施 |
|------|----------|
| 时间不够，功能做不完 | 砍掉非核心功能，优先保证预约主流程闭环 |
| 页面越做越多，超出控制 | 严格控制页面数量，超出 12 页需砍掉低优先级页面 |
| SQL Server LocalDB 环境配置出问题 | 提前验证开发环境，准备 Docker / SQLite 备选方案 |
| AI 生成的代码不稳定、逻辑有坑 | 每段代码生成后手动走读核心逻辑，重点测试边界情况 |
| 前后端耦合太紧，改一处崩一片 | 严格分层 Controllers / Services / Repositories |
| Git 操作失误丢代码 | 每次重要节点手动 commit，不等到最后一次性提交 |
