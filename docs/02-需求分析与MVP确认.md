# 图书馆座位预约系统 —— 需求分析与 MVP 确认

## 1. 目标用户与核心场景

### 用户分类

| 用户类型 | 身份 | 技术能力 | 使用频率 | 核心诉求 |
|----------|------|----------|----------|----------|
| 普通学生 | 在校本科生/研究生 | 基本网页操作 | 每日/隔日 | 快速找到空座并预约 |
| 管理员 | 图书馆工作人员 | 基础计算机操作 | 每日 | 管理座位和预约记录 |

### 核心场景

**场景一：学生预约座位**
> 周四晚上，小王打算明天去图书馆自习。他打开系统，看到三楼还有空位，选了一个靠窗的座位，预约了明天 9:00-12:00 的时间段。系统提示预约成功。

**场景二：学生取消预约**
> 第二天早上小王起晚了，去不了图书馆。他打开"我的预约"，取消了今天的预约，座位被释放给其他人。

**场景三：管理员管理座位**
> 图书馆工作人员发现 3 楼 12 号座位的灯管坏了，在后台将该座位标记为"维修中"，避免学生误约。

**场景四：管理员查看统计**
> 期末临近，馆长想知道本周座位利用率。管理员登录后台查看统计页，看到本周平均利用率达到 85%。

## 2. 用户最关心的问题

### 学生关心
1. 到了图书馆才跑空？→ **提前查空座、线上预约**
2. 预约操作麻烦吗？→ **选座 + 选时段 + 提交，三步完成**
3. 约了去不了怎么办？→ **随时取消，释放座位**
4. 怎么证明我约了？→ **我的预约里有记录可查**

### 管理员关心
1. 哪些座位被占了？→ **可视化座位状态总览**
2. 有人恶意占座不去？→ **查看预约记录，手动取消违规**
3. 座位坏了怎么办？→ **后台标记维修/关闭**
4. 整体使用情况如何？→ **统计页看数据**

## 4. 功能优先级矩阵

### 核心基础（P0 — 不做则主流程断裂）

这些功能必须在第一轮开发中完成，否则用户无法完成"查座→预约→取消"这个基本闭环。

| 编号 | 功能 | 理由 |
|------|------|------|
| F01 | 体验账号切换 | 没有身份识别就无法区分是谁的预约 |
| F03 | 座位列表 | 查座是整个流程的起点 |
| F05 | 座位详情 | 不看详情没法决定约不约 |
| F06 | 预约时段选择 | 预约的核心输入 |
| F07 | 预约提交（含冲突检测） | 核心动作 |
| F08 | 预约成功提示 | 提交后必须给反馈 |
| F09 | 我的预约 | 预约后需要能看到记录 |
| F10 | 取消预约 | 订了去不了必须能释放 |
| F16 | Seed 数据初始化 | 没有数据页面全是空的，无法演示 |

### 主体业务（P1 — 必须完成，完善系统闭环）

这些功能将系统从"一个预约表单"升级为"一个可管理的信息系统"。

| 编号 | 功能 | 理由 |
|------|------|------|
| F02 | 首页 | 系统入口，公告和导航 |
| F11 | 管理员登录 | 必须有入口保护后台 |
| F12 | 管理员预约管理 | 管理员核心职责 |
| F13 | 管理员座位管理 | 维护座位数据 |
| F14 | 统计页 | 立项单明确要求 |
| F15 | 404 错误页 | 基础体验保障 |

### 体验提升（P2 — 有更好，不做不影响功能可用）

| 编号 | 功能 | 理由 |
|------|------|------|
| F04 | 按区域筛选 | 默认用分页/分区域 tab 替代，不作为独立筛选项强制开发 |
| F19 | 预约历史归档 | 当前仅显示"当前预约+历史预约"同页即可，不用单独做归档 |

### 暂缓（P3 — 本期明确不做）

| 编号 | 功能 | 理由 |
|------|------|------|
| F17 | 座位收藏/偏好 | 与主流程无关，后续迭代 |
| F18 | 违规记录黑名单 | 需要真实账号体系支撑 |
| F20 | 座位评论/评分 | 需要用户生成内容管理 |

## 5. MVP 范围确认

### MVP 边界

**包含（P0 + P1，共 15 项功能 / 9 个页面）：**

用户端（4 页）：
- 首页、座位列表、座位详情（含预约表单，提交后 TempData 提示并跳转至"我的预约"）、我的预约

管理端（4 页）：
- 管理员登录、预约管理、座位管理、统计页

通用 + 共享：
- 体验账号切换（布局级组件）、404 错误页、Seed 数据

**不包含：**
- F04 按区域筛选（默认用 tab 分区域即可，不需要单独开发）
- F17/F18/F20 等暂缓功能
- 立项单中列出的全部暂不开发内容

### MVP 总工作量估算

| 维度 | 估算 |
|------|------|
| Controller | 约 4~5 个（Home, Seat, Reservation, Admin/Login, Admin/Manage） |
| View | 9 个页面 + 1 个布局页 |
| Model / Entity | 约 3~4 个（User, Seat, Reservation + ViewModel） |
| 数据库表 | 约 3 张（Seats, Reservations, Users 预置体验账号） |
| 预计开发周期 | 2 周核心开发 + 1 周联调修复 |

## 8. 进入 PRD Lite 前的最终口径

在开始写 PRD Lite 之前，确认以下边界：

**做什么：**
一个图书馆座位预约系统的 Web 应用，学生能在线上查座 → 选座 → 预约 → 取消，管理员能管理座位和预约记录，提供基础统计数据。采用体验账号切换代替真实注册登录。

**不做什么：**
不做支付、扫码、消息推送、真实账号体系、硬件对接、多馆区。总共 9 个页面，4 周交付。

**第一优先级（启动就做）：**
预约核心闭环 —— 体验账号切换 → 查座 → 详情 → 预约 → 我的预约 → 取消。

**第二优先级（核心闭环后做）：**
管理后台 —— 管理员登录 → 座位管理 → 预约管理 → 统计页。

**边界清晰点：**
- "预约提交"不是独立页面，是座位详情页内的表单 POST
- 体验账号切换在导航栏实现，不单独占一个页面
- 统计页只展示基础数字和简单图表，不做复杂数据可视化
