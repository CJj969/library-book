## C. 必改项

### P1：原型评审尚未执行，清单是空壳

**位置：** prototype/review-1/原型评审清单.md

**现状：**
- 第 1~4 节：所有 25 个检查项全部标记 ⚠️（未评），没有一个是 ✅ 或 ❌
- 第 5 节（必须在接后端前修改的问题）：4 行全部空白
- 评审结论：全部"待评估" / "待填写"

**问题：**  
这不是"已完成原型评审"，只是"建了一份评审模板"。评审清单的意义是逐项过检、记录结论、汇总必改项。当前状态意味着：如果你现在拿着这个原型去进入系统设计，开发者的第一句话就是"你们评审过了吗？改什么？"——答不上来。

**修正方案：**

花 30 分钟逐项执行评审，按以下规则填写：
- ✅ = 通过，🟡 = 轻微问题但不阻塞，❌ = 必须修改

### P2：部分管理端页面未使用自定义组件体系

**位置：** admin-login.html、admin-statistics.html

**问题：**  
style.css 定义了完整的组件体系（.btn-seat-* / .form-control-seat / .alert-seat-* / .stat-card-admin 等），但：
- `admin-login.html`：使用 `btn btn-primary`、`form-control`、`alert` 等 Bootstrap 默认类
- `admin-statistics.html`：使用 Bootstrap 默认卡片 + `shadow-sm`，而不是 `.stat-card-admin`；统计数字直接写 `display-5 fw-bold` 而不是用 `.stat-number`

这会带来两个实际问题：
1. 后端开发人员做 Razor View 时不知道该用哪个类名体系
2. 页面视觉不统一（登录页的按钮圆角、字号与会者不一致）

**修正方案：**

统一替换为 style.css 的自定义类：

`admin-login.html`：
- `<input class="form-control">` → `<input class="form-control-seat">`
- `<button class="btn btn-primary w-100">` → `<button class="btn-seat btn-seat-primary btn-seat-block" style="height:44px;">`
- `<div class="alert">` → `<div class="alert-seat alert-seat-danger">`（注意目前 alert 内已有 alert-seat-danger，但外层包装不一致）

`admin-statistics.html`：
- 统计卡片改用 `.stat-card-admin` + `.stat-card-admin.stat-*` 类
- 导航栏改用 `.admin-nav` 类（见 O1）
- 移除 `<main class="container-fluid">`，改用 `.admin-layout > .page-content-wide`

## D. 建议优化项

### O1：admin-statistics.html 导航栏统一为 .admin-nav

当前使用 `<nav class="admin-nav navbar navbar-expand-lg">` + `container-fluid`，而其他管理端页面使用纯 `<header class="admin-nav">`。除了视觉差异外，还会让后端 Razor 开发时不确定管理端导航栏的 HTML 结构。

建议直接对齐到 `admin-reservations.html` 的导航栏 HTML 结构。

### O2：修复 admin-seats.html 的死链接

第 18 行 `<a href="./admin-stats.html">统计页</a>`，实际文件名为 `admin-statistics.html`。点击会 404。

### O3：404.html 导航栏统一为 .mobile-nav

404.html 使用 `<nav class="navbar navbar-expand bg-white border-bottom">`，而用户端其他页面使用 `<header class="mobile-nav">`。虽然 404 不是高频页面，但风格分裂会让人感觉是"两个系统拼出来的"。
